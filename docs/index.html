kitz 🔱

Capture Claude Code commands, skills & plugins — on the fly.

You keep telling Claude the same thing and think "this should be a /command." But stopping to recall the file format, the right folder, and writing the prompt kills the impulse. kitz collapses that to one word — capture the idea, or just describe it and let Claude write the whole thing.

What it is

A Claude Code /command or skill is just a markdown file in a .claude/ folder. kitz writes that file for you: pick a name, pick where it goes, done. One engine, three muscle-memory names — the name picks the type, so there's no extra step.

cmdz
a slash command
.claude/commands/<name>.md → /name
sklz
a skill (auto-triggered)
.claude/skills/<name>/SKILL.md
plgz
a plugin scaffold
.claude/plugins/<name>/

Bare kitz is the home dashboard — run it in a terminal and you get one cross-type list of everything you've made (commands, skills, plugins, every scope) with a live preview. enter edits · ctrl-y copies · ctrl-o reveals it in Finder · ctrl-x deletes · ctrl-n makes a new one · ctrl-v makes one from your clipboard · ctrl-g drafts one with Claude. Copying or revealing flashes a ✓ confirmation in the header, and the preview scrolls with shift-↑/↓ or PageUp/PageDown (or the trackpad). In a script or pipe it stays a plain umbrella (kitz --doctor / --help / --version), so nothing that shells out to it breaks.

Install

Option A · Homebrew shareable

The clean, reinstallable path once the tap is published to GitHub:

brew
brew install h0bb5/tap/kitz   # installs kitz, cmdz, sklz, plgz
One-time publish first. Uses one shared tap (h0bb5/homebrew-tap) for all your tools. Push this repo + tag it (git tag v0.1.1 && git push --tags), run ./release.sh v0.1.1 to get the url/sha256, paste those into homebrew/kitz.rb, and push that file to h0bb5/homebrew-tap at Formula/kitz.rb. Until then, use Option B.

Option B · Local works right now

Symlinks the four commands into your PATH. This is already done on this machine.

sh
sh ~/Projects/kitz/install.sh        # re-run any time; idempotent
sh ~/Projects/kitz/install.sh --uninstall

Needs fzf (pickers). bat (preview), $EDITOR, and claude (for the ghostwriter) are optional.

Test it on this machine

1

Confirm your setup

kitz --doctor
kitz --doctor   # deps, editor, model, and exactly which folder each scope writes to
2

Preview a command (writes nothing)

The -n dry-run shows the file it would create.

cmdz
cmdz hello -m 'Say hi to $ARGUMENTS' -n
3

Actually save one

Drop -n, add a scope and -y (no prompts).

cmdz
cmdz hello -m 'Say hi to $ARGUMENTS' -s user -y
# → ~/.claude/commands/hello.md
4

Let Claude write one ✦

Describe it; Claude drafts the whole file in ~10s, then it opens in your editor to review.

cmdz -g
cmdz standup -g -i "summarize yesterday from git log, list today's plan by priority"
5

Use it in Claude

New commands appear after a session reload (/clear or a fresh session). Then:

claude
/hello world
/standup
Inside a Claude Code session? Prefix any of the above with ! to run it in your shell (e.g. !cmdz hello -m '…' -n). Handy, but see the honest note under Triggers — the terminal isn't where kitz earns its keep.

Two modes, three speeds

SpeedCommandPrompts
One-shotcmdz x -m '…' -s project -y0
Quickcmdz xeditor opens → pick scope
Ghostwritecmdz x -g -i '…'Claude drafts → refine → save

In ghostwrite mode the refine loop lets you regenerate or tweak with a note (Claude revises its own draft) with a live preview, before you commit. Non-interactive variant: --revise "make it shorter".

Run cmdz / sklz / plgz with no name and you get a browser of what already exists, with a live preview. In it: enter edits · ctrl-y copies the file's contents to your clipboard · ctrl-o reveals it in Finder · ctrl-x deletes it (skills/plugins remove the whole dir; asks first). Modifier keys, not bare c/x — typing filters the list.

Triggers — the real flow

The point of kitz isn't the terminal — it's capturing a command from anywhere, when you're not already in a Claude session. Snippets live in ~/Projects/kitz/integrations/.

tmux prefix + G = ghostwrite popup

~/.tmux.conf
# add these, then: tmux source-file ~/.tmux.conf
bind-key C display-popup -E -w 80% -h 80% "cmdz"    # capture a command
bind-key G display-popup -E -w 80% -h 80% "cmdz -g" # Claude drafts it

nvim :KitzGen — selection becomes the brief

init.lua
require('kitz').setup()   # after copying integrations/nvim/kitz.lua onto your runtimepath
# :KitzGen  → ghostwrite · visual-select a description, :KitzGen → Claude drafts it

Raycast keystroke from anywhere

Copy integrations/raycast/*.sh into your Raycast script-commands folder (~/Documents/RaycastScripts). "Draft Claude Artifact (AI)" takes a name + brief + scope and Claude writes it.

The honest bit. Running kitz with ! inside Claude Code is barely faster than just asking Claude "make me a /standup command." Its reason to exist is the triggers above — a command spun up in the 5 seconds before the idea evaporates, without context-switching into a session. That's the part worth testing. If prefix+G fits your muscle memory, kitz was worth building. If it doesn't, it wasn't.
